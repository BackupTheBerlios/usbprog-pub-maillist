Am Donnerstag, 4. Oktober 2007 schrieb Benedikt Sauter:
> Hello Sven,
>
> >As I have already mentioned in the forum, I am very much interested in
> >starting development of an XSVF player for usbprog. I am currently having
> > a look at some of the usbprog code in the svn and also at the XSVF
> > Executor code from the Ethernut project. Up to now I think it shouldn't
> > be too difficult to port the Ethernut code to usbprog. I have made up
> > some ideas:
> >
> >As XSVF instructions can become quite long I think it's best to send one
> >instruction at a time via usb. This way usbprog can also send back
> >information to the player software running on the computer wether the
> > current XSVF instruction was successful or not.
>
> Yes you can use the simpleport firmware as draft.
>
> >Is it correct that a USB packet has a maximum payload size of 1023 bytes
> > in our case, so that XSVF instructions of nearly 1kB could be supported
> > without fragmentation? I think that should be enough for the beginning,
> > as we only have 2kB static RAM anyway.
>
> This is only the theoretical rate. The max packet size with USBN9604 is
> 64 Byte. It is possible so send with the libusb and the atmega32 on  the
> otherside max 3 packets in a USB Frame (1 ms), but there we need some
> further hacks.
>
> The easiest way for the first version is that you use 64 Byte. Because
> than you can work 1:1 like in simpleport. If you are on an USB 2.0
> Interface after an HUB the usb stack also transfer 8x 64 Byte per
> Second. Without extra modifying anything!
>
> I can try to build easy functions for a bigger transfer but the max is
> about  200 KB/s. But actual I have to much work. When you can start with
> a 64 Byte version I would be very pleasure :-)
>
> Is that ok?

Yes, that's no problem, and thank you for the information. I've created some 
code yesterday to split an entire XSVF file into seperate XSVF instructions 
in order to see how long they become. I tested with two different XSVF files, 
created for programming a single Xilinx XC9572 CPLD in the JTAG chain. The 
resulting XSVF instructions had a maximum length of 11 bytes, so I think 64 
bytes maximum should be more than enough for the beginning.
